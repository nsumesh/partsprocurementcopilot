# Design Decisions

Living document. Each entry: what was chosen, what was considered, and why the choice wins.
Update this file whenever a decision is made or reversed.

---

## Data

### OE Catalog Source
**Chosen:** Browserbase — headless browser-as-a-service to scrape finditparts.com

**Alternatives considered:**
- Direct HTTP + httpx/BeautifulSoup — free and fast, but finditparts.com requires JS rendering and blocks bots
- ScraperAPI — cheaper, but less control over complex sessions and weaker anti-bot handling

**Why:** Browserbase handles JS rendering, rotating residential proxies, and session management without managing browser infra locally. finditparts.com is a JS-heavy site; raw HTTP scraping fails. Browserbase is the reliable path.

---

### Aftermarket Data Source
**Chosen:** LLM-generated synthetic CSV — aftermarket SKUs generated by LLM after OE catalog extraction, representing third-party/generic equivalents for the same part categories. Stored as CSV then ingested alongside OE data.

**Alternatives considered:**
- Real aftermarket API (e.g., PartsTech, Epicor) — accurate and up-to-date, but requires paid access and API integration complexity outside scope
- Hand-written fixture CSV — fast to create, but tedious and doesn't scale with OE catalog size

**Why:** Synthetic generation is fast, scales with whatever OE categories are scraped, and is sufficient for eval and demo purposes. No paid API dependency. LLM can produce realistic SKUs, brand names, and pricing that mirror real aftermarket patterns.

## Search

### Hybrid Retrieval Strategy
**Chosen:** SQLite FTS5 (BM25) for keyword search + Supabase pgvector (ANN) for semantic search, merged via RRF (k=60).

**Alternatives considered:**
- ParadeDB `pg_search` — true BM25 inside Postgres, but requires Supabase paid tier ($25/mo)
- `pg_trgm` — free Postgres trigram similarity, but not true BM25; weaker keyword precision
- Elasticsearch — real BM25, but paid on Elastic Cloud or self-hosted complexity; requires separate data sync outside Postgres

**Why:** SQLite FTS5 provides genuine BM25 ranking with zero infra cost — it runs in-process alongside the API. pgvector on Supabase free tier handles semantic search. RRF normalises the two ranked lists without needing score calibration. The only operational concern is Railway's ephemeral filesystem: the SQLite index must either be persisted via a Railway volume or rebuilt from Supabase on startup (fast since source data is already in Postgres).

## Async Architecture

### Task Execution and Streaming
**Chosen:** FastAPI + asyncio — pipeline runs directly in the async request handler, results streamed to the client via SSE as each step resolves. No task queue, no broker, no worker process.

**Alternatives considered:**
- Celery + Redis — distributed task queue with `task_id` pattern; handles retries and scale, but adds two extra Railway services (worker + Redis) and significant boilerplate for a ~1000-user tool
- ARQ — lightweight async Redis queue, simpler than Celery but still requires a broker and separate worker process

**Why:** At ~1000 users, in-process async is sufficient. FastAPI's async request handlers keep the pipeline non-blocking. SSE streams each ranked part directly from the handler as it resolves — no intermediate queue or polling needed. Eliminates Redis, Celery worker, and task-id plumbing entirely. Simpler to deploy (single Railway service for the API).

## AI Pipeline

### Fitment Assignment
**Chosen:** LLM (claude-sonnet-4-6) at temperature 0 for fitment confidence classification. Structured `fit_notes` match runs first; LLM called only when structured match is absent or ambiguous.

**Alternatives considered:**
- Higher temperature — introduces non-determinism in a classification task where the output is a fixed enum (High / Medium / Low / No Fitment); no benefit
- Rule-only (no LLM fallback) — fast and cheap, but breaks on ambiguous or incomplete fit_notes

**Why:** Temperature 0 is correct for a structured classification task with a fixed output schema. The LLM must return one of four confidence levels plus a reasoning string — determinism is a requirement, not a preference. Structured match handles the easy cases cheaply; LLM handles edge cases accurately.

---

### Embeddings Provider
**Chosen:** Cohere `embed-english-v3.0` — used at ingest (document input type) and at query time (query input type).

**Why:** Cohere's v3 embeddings are strong on asymmetric retrieval (short query vs. long document), which matches this use case. Separating query vs. document input types improves retrieval quality with no extra cost.

---

### Reranker
**Chosen:** Cohere `rerank-english-v3.0` — cross-encoder applied to the merged RRF candidate set before fitment assignment.

**Why:** Cohere Rerank is purpose-built for this step, has a generous free/trial tier, and requires no self-hosting. Cross-encoder scoring at rerank time is more accurate than bi-encoder similarity alone.

---

### Intent Parsing
**Chosen:** claude-sonnet-4-6 at temperature 0 — extracts part category, attributes, and emits a clarifying question when the query is ambiguous.

**Why:** Same model and temperature rationale as fitment assignment: structured JSON output with a fixed schema requires determinism. Clarifying question logic must be consistent across identical inputs.

## Data Infrastructure

### VIN Decoding
**Chosen:** NHTSA VPIC API as primary (free, public, covers all US commercial truck VINs). Pre-decoded local records for the 5 eval VINs seeded at ingestion time as fallback.

**Why:** NHTSA VPIC is free and authoritative for US VINs. Local fallback ensures eval queries never fail due to network issues or API downtime.

---

## Deployment

### Infrastructure
**Chosen:** Railway — API service and frontend service deployed as separate Railway services. Supabase for Postgres + pgvector (remote). SQLite index lives on the API service filesystem, rebuilt from Supabase on startup.

**Why:** Railway handles containerised deploys with minimal config. Two services (API + frontend) keeps concerns separated without the overhead of a third worker service. Supabase free tier covers the database.

---

## Product

### Urgency Toggle
**Chosen:** Urgent mode routes to a higher-priority model path targeting lower latency. Standard uses the default pipeline.

**Why:** Urgency is a real signal from the operator — it should actively change pipeline behavior, not just UI styling. Faster results when urgency is high is the correct product response.

---

### Orders
**Chosen:** Intent records only — order confirmation persists a request record to the database. No payment processing or external purchasing integration.

**Why:** The tool is a procurement copilot, not a purchasing system. Recording intent is sufficient for fleet operators to act on results externally.

---

### Eval Runner
**Chosen:** CLI script only (`uv run python -m ingestion.eval_runner`). Runs all 5 golden queries through the live pipeline and prints per-query metrics.

**Why:** Evals are a developer tool, not an operator-facing feature. A CLI is sufficient and avoids building and maintaining an admin UI.

---

### Authentication
**Chosen:** Single-user, no authentication. The tool is accessed directly without login.

**Why:** Scope is a single-operator tool. Auth adds infra and UX complexity with no benefit at this scale.

## Frontend — UI Design System

### Color Theme: Black & Orange
**Chosen:** Dark theme (`zinc-950` background, `zinc-900` cards, `zinc-800` inputs) with `orange-500` as the sole primary accent. Text hierarchy: `white` → `zinc-400` → `zinc-500` → `zinc-600`.

**Why:** Fleet operators use procurement tools in workshops and depots where screen glare is high and ambient light is variable. High-contrast dark UIs reduce eye strain. Orange is an industrial/safety color strongly associated with heavy equipment and fleet operations (Caterpillar, JCB, Liebherr). Orange CTAs on near-black surfaces achieve the highest contrast ratio for primary actions.

---

### Component Scale
**Chosen:** Larger inputs (`py-3 text-base`), larger cards (`px-6 py-5`), `text-lg` part names, `text-xl` prices.

**Why:** Fleet operators commonly access tools on shop tablets or with work gloves on. Larger tap targets and higher-density typography reduce input errors. "Bigger" is a better default for industrial tooling than the web-app norm.

---

### Branding: HeaviAI Procurement CoPilot
**Chosen:** "HeaviAI" as product family, "Procurement CoPilot" as module name. "Procurement" highlighted in orange on the landing hero. "CoPilot" capitalized mid-word (cf. GitHub CoPilot convention).

**Why:** Establishes the product line (HeaviAI) separate from the feature (Procurement CoPilot), allowing other CoPilot modules (e.g., Maintenance CoPilot, Compliance CoPilot) to be added later under the same brand.

---

### Two-Screen Flow
**Chosen:** `/` is a landing form (VIN + query + urgency); submit navigates to `/results` where streaming begins. Router state carries `{ vin, query, urgency }`.

**Why:** Separating the search form from the results page gives each screen a single job. The landing screen can be bookmarked and shared. The results screen starts streaming immediately on mount with a full-screen layout optimized for scanning results. Mixing both states on one page creates an awkward collapsing layout as results arrive.

---

### Filters: Client-Side Only
**Chosen:** Filters (price, year range, source, fitment) are applied in the browser against the already-streamed `SearchResultPart[]` array. No additional API calls on filter change.

**Why:** The retrieval pipeline returns at most 10 results. Client-side filtering of 10 items is instant and avoids round-trips. Adding server-side filter params would require re-triggering the full AI pipeline (intent → embed → retrieve → rerank → fitment), which is expensive and slow. The correct pattern is to retrieve broadly then filter locally.

---

### Deployment: Railway Monorepo — Per-Service railway.toml

**Chosen:** One `railway.toml` per service directory (`backend/railway.toml`, `frontend/railway.toml`), each containing only `[build]` and `[deploy]` sections. No root-level multi-service config.

**Why:** Railway config-as-code only supports top-level `[build]` and `[deploy]` for a single service. `[[services]]` blocks are not a valid Railway construct — using them caused Railway to ignore `dockerfilePath` and fall back to Railpack auto-detection, which failed because no recognizable app existed at the repo root. Per-directory configs with Root Directory set per service in the dashboard is the correct Railway monorepo pattern.

---

### Deployment: Dynamic Port via $PORT

**Chosen:** Both backend and frontend read `$PORT` at runtime. Backend uses shell-form CMD with `${PORT:-8000}` fallback. Frontend uses `envsubst` to substitute `${PORT}` into the nginx config template at container startup.

**Why:** Railway assigns a dynamic port via `$PORT` and routes all traffic (including healthchecks) to it. Hardcoding `8000` or `80` means the app starts but is unreachable — healthchecks fail and Railway marks the deployment as failed. The `:-8000` / `:-80` fallbacks preserve local development without any env setup. nginx requires `envsubst` because it does not natively expand environment variables in config files; `envsubst` must be scoped to `${PORT}` only to avoid clobbering nginx's own `$uri`, `$host`, and other internal variables.

---

### Deployment: VITE_API_BASE_URL Must Be Set Before Build

**Chosen:** `VITE_API_BASE_URL` is set as a Railway environment variable on the frontend service before the first `railway up` or GitHub-triggered build.

**Why:** Vite bakes `import.meta.env.VITE_*` values into the JS bundle at build time — they are not runtime environment variables. If the variable is missing or set to `localhost:8000` at build time, every user's browser makes API calls to localhost (which fails silently). The correct order is: deploy backend → get its domain → set `VITE_API_BASE_URL` → deploy frontend.
